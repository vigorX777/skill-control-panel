# Skill Control Panel 独立化 MVP 功能清单

## 1. 文档目的

这份文档用于定义 `skill-control-panel` 从“依赖外部 `asm`”改造成“可独立运行的本地技能管理面板”的 MVP 范围。

目标是先确认最小可用能力，不追求一次性替代完整 `asm`。

---

## 2. MVP 目标

MVP 版本需要满足以下目标：

- 不依赖外部 `asm` 命令即可运行
- 保留当前控制台最核心的管理能力
- 尽量复用现有前端和 API 结构，降低改造成本
- 优先支持日常技能管理，而不是高级运维能力

---

## 3. MVP 范围定义

### 3.1 必须保留的用户价值

- 查看本地已安装技能
- 按分类、状态、Provider 等方式筛选技能
- 识别重复技能
- 启用或禁用 Provider
- 安装技能
- 卸载技能
- 编辑技能管理元数据

### 3.2 明确不纳入 MVP 的内容

- 完整复刻 `asm` 全部 CLI 能力
- 自动修复重复技能
- `bundle`、`import`、`link`、`index` 等高级能力
- 技能内容编辑器
- 自动迁移现有 `asm` 配置
- 复杂来源安装协议
- 远程同步和发布能力

---

## 4. 功能清单

以下条目是本次独立化 MVP 需要逐项确认的功能需求。

### F01. Provider 配置管理

**目标**

系统内部维护 Provider 注册表与启用状态，不再通过 `asm config` 获取配置。

**需求**

- 支持读取本地 Provider 配置
- 支持保存 Provider 配置
- 每个 Provider 至少包含以下字段：
  - `name`
  - `label`
  - `globalPath`
  - `projectPath`
  - `enabled`
- 默认内置一组常用 Provider
- 前端可以读取 Provider 列表并展示启用状态

**验收标准**

- 启动服务后，即使系统未安装 `asm`，也能正常返回 Provider 列表
- 在控制台切换 Provider 启用状态后，刷新后状态仍然保留

**确认状态**

- [x] 已确认

### F02. 技能扫描

**目标**

系统自行扫描各 Provider 目录，构建技能清单。

**需求**

- 支持扫描全局技能目录
- 支持扫描当前项目技能目录
- 识别技能基础信息：
  - `name`
  - `description`
  - `version`
  - `dirName`
  - `path`
  - `realPath`
  - `provider`
  - `providerLabel`
  - `scope`
- 能识别软链接与真实路径
- 输出结构尽量兼容现有前端使用的数据格式

**验收标准**

- 在不依赖 `asm` 的情况下，首页可以正常展示技能列表
- 同一个技能若出现在多个 Provider 下，前端仍能正确显示 Provider 归属

**确认状态**

- [x] 已确认

### F03. 技能管理元数据

**目标**

保留当前控制台对技能的附加管理信息。

**需求**

- 保留并支持编辑以下字段：
  - `category`
  - `priority`
  - `status`
  - `preferredProvider`
  - `notes`
- 元数据独立持久化，不写回 `SKILL.md`
- 对没有人工标注的技能，自动生成默认分类、状态、优先级

**验收标准**

- 用户修改元数据后，刷新页面仍能看到更新结果
- 重新扫描后，不会丢失已保存的元数据

**确认状态**

- [x] 已确认

### F04. 重复技能检测

**目标**

保留当前“重复分析”能力，用于识别多 Provider 暴露和真实副本重复。

**需求**

- 识别“同一真实路径被多个 Provider 暴露”的重复
- 识别“不同真实路径但技能名或目录名重复”的重复
- 生成重复分组数据
- 为每个技能打上重复模式标记

**验收标准**

- 前端可以继续显示“重复分组”
- 同一真实路径的重复项和不同真实路径的重复项能被区分

**确认状态**

- [x] 已确认

### F05. Provider 启用/禁用

**目标**

在控制台中直接管理 Provider 是否参与使用。

**需求**

- 支持将 Provider 启用/禁用变更加入变更篮
- 应用后持久化到本地配置
- 被禁用 Provider 的状态可在 UI 中明确展示

**验收标准**

- 用户执行启用/禁用后，配置文件同步更新
- 刷新后前端状态与配置文件一致

**确认状态**

- [x] 已确认

### F06. 技能安装

**目标**

在不依赖 `asm install` 的前提下完成技能安装。

**需求**

- 支持从指定来源安装技能
- MVP 安装来源暂定支持：
  - GitHub 来源
  - 本地路径来源
- 支持指定目标 Provider
- 安装完成后自动重新扫描

**验收标准**

- 用户可以从控制台发起安装
- 安装成功后，技能会出现在列表中
- 安装失败时，结果区能看到错误信息

**确认状态**

- [x] 已确认

### F07. 技能卸载

**目标**

在不依赖 `asm uninstall` 的前提下完成技能卸载。

**需求**

- 支持把卸载动作加入变更篮
- 支持卸载具体技能实例
- 卸载后自动重新扫描

**验收标准**

- 用户执行卸载后，目标技能实例从列表中消失
- 卸载失败时，结果区能看到错误信息

**确认状态**

- [x] 已确认

### F08. 本地服务与前端兼容

**目标**

尽量保持现有服务和前端结构稳定，优先替换后端能力来源。

**需求**

- 保留现有本地 HTTP 服务模式
- 保留现有 dashboard 页面结构
- 保留现有主要 API 路由，尽量避免前端大改
- 保留变更篮机制

**验收标准**

- 页面主要交互无需重做即可继续使用
- 改造后前端仍能完成筛选、分组、详情查看和变更应用

**确认状态**

- [x] 已确认

## 5. 待确认的产品决策

以下不是实现细节，而是 MVP 需要先定下来的范围问题。

本节先给出建议默认值，便于快速确认。

### D01. Provider 范围

当前建议：

- MVP 内置固定 Provider：
  - `claude`
  - `codex`
  - `agents`
  - `opencode`

建议理由：

- 这 4 个 Provider 已经覆盖当前 skill 的主要使用场景
- 可以显著降低独立化后的扫描、安装和卸载复杂度
- 数据模型仍保留可扩展性，后续可以再补自定义 Provider

待确认：

- [x] 只支持以上 4 个 Provider 作为 MVP 默认集

### D02. 安装来源范围

当前建议：

- MVP 支持：
  - `github:user/repo`
  - 本地路径

建议理由：

- GitHub 来源可以覆盖大多数技能安装场景
- 本地路径便于开发期调试和私有技能接入
- 先不做更多来源协议，能明显降低安装器实现成本

待确认：

- [x] 先只做这两种安装来源

### D03. 扫描范围

当前建议：

- 扫描全局目录
- 扫描当前项目目录

建议理由：

- 这是最符合用户直觉的默认扫描范围
- 足够覆盖“全局技能”和“项目专属技能”两类核心场景
- 先不开放额外自定义目录，能减少配置复杂度和边界情况

待确认：

- [x] 暂不支持额外自定义扫描目录

### D04. 卸载粒度

当前建议：

- 优先支持“实例级卸载”
- 如有必要，后续再补“按技能名卸载所有实例”

建议理由：

- 实例级卸载更安全，不容易误删其他 Provider 下的同名技能
- 更符合多 Provider 模型
- 后续若要加“按技能名卸载所有实例”，可以作为增强功能补上

待确认：

- [x] 将实例级卸载作为 MVP 默认行为

---

## 5.1 建议确认稿

如果没有特殊诉求，建议先按以下默认决策推进 MVP：

- D01：MVP 默认只支持 `claude`、`codex`、`agents`、`opencode`
- D02：MVP 仅支持 `github:user/repo` 和本地路径安装
- D03：MVP 仅扫描全局目录和当前项目目录
- D04：MVP 默认采用实例级卸载

---

## 6. 非功能性要求

- 改造后不要求用户额外安装 `asm`
- 本地数据文件需要可读、可维护
- 扫描与安装失败时必须返回清晰错误
- 现有测试结构应可继续扩展
- 优先保证行为稳定，再考虑抽象完美

---

## 7. 建议的 MVP 完成定义

满足以下条件即可视为 MVP 完成：

- 系统未安装 `asm` 时仍可正常启动控制台
- 能列出技能、Provider、重复组
- 能修改技能元数据
- 能启用/禁用 Provider
- 能安装技能
- 能卸载技能
- 前端主界面无需大规模重写

---

## 8. 下一步

建议下一步按以下顺序继续推进：

1. 确认第 5 节中的 4 个产品决策
2. 将第 4 节功能清单逐项确认
3. 基于确认结果输出改造实施方案
4. 分阶段替换现有 `asm` 依赖层
